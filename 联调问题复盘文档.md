# 联调问题复盘文档

## 1. 文档目的

本文档专门复盘第二阶段本地联调时发现并修复的两个问题：

1. 下载时找不到真实文件
2. 同路径重复上传时客户端卡住

这两个问题都不是“编译不过”那种静态错误，而是只有真正跑通客户端、服务端、数据库、磁盘落盘这整条链路之后才会暴露出来的联调问题。

所以这份文档的重点不是简单写一句“改了哪里”，而是把下面这些内容完整讲清楚：

1. 问题出现时，外部现象是什么
2. 为什么代码表面上看起来没问题，但运行时却会失败
3. 根因到底在哪一层
4. 最后具体改了什么
5. 以后怎么避免类似问题

---

## 2. 问题一：下载时找不到真实文件

## 2.1 问题现象

在本地联调时，上传和秒传都已经成功：

1. 用户 `stage2e` 首次上传成功
2. 用户 `stage2f` 上传同内容文件时秒传成功
3. 数据库中 `paths` 和 `files` 记录也都正常存在

但是当 `stage2e` 执行下载命令：

```text
gets windcloud_stage2_upload.txt
```

客户端却收到：

```text
服务器文件不存在
```

也就是说：

- 从用户视角看，文件明明已经上传成功
- 从数据库视角看，`paths` 和 `files` 也都存在
- 但服务端在下载阶段却说“找不到”

这说明问题不在“有没有这条逻辑文件记录”，而是在“服务端最终去磁盘上找真实文件”这一步。

---

## 2.2 当时的直接排查思路

遇到这种问题，不能只看客户端提示，因为客户端看到的只是“服务端说没有这个文件”。

真正要查的是这条完整链路：

```text
客户端文件名
-> 当前虚拟路径
-> paths.file_id
-> files.sha256sum
-> 真实磁盘路径
-> open()
```

所以排查时，我做了两件事：

### 第一步：查数据库

联调时查询了数据库：

```sql
SELECT id, user_id, path, file_id, parent_id, file_name, type FROM paths;
SELECT id, HEX(sha256sum), size, count FROM files;
```

结果表明：

1. `paths` 表中存在：

```text
/windcloud_stage2_upload.txt -> file_id = 1
```

2. `files` 表中存在：

```text
id = 1
sha256sum = 86D82221F388C7A0...
size = 58
count = 2
```

数据库记录本身没问题。

### 第二步：查服务端日志

服务端日志中出现了关键错误：

```text
打开真实文件失败，路径=./test/files/86D82221F388C7A0B3933A6501013967E5B07B21A65CB9B9ED0487D45DF1A446，错误码=2
```

而真实磁盘上的文件名实际是：

```text
./test/files/86d82221f388c7a0b3933a6501013967e5b07b21a65cb9b9ed0487d45df1a446
```

日志一下就把问题定位出来了：

- 数据库取出来的是大写 hash
- 磁盘落盘时保存的是小写 hash

Linux 文件名是区分大小写的。

所以：

```text
86D8...
```

和

```text
86d8...
```

对 Linux 来说是两个完全不同的文件名。

---

## 2.3 根因分析

这个问题的根因不在上传，也不在 `paths` 表，而在 `files -> 真实文件路径` 这一跳上。

### 2.3.1 上传时真实文件名的来源

上传时，客户端算出来的 SHA-256 是通过 `sha256sum` 命令得到的，默认是小写十六进制字符串，例如：

```text
86d82221f388c7a0...
```

服务端正常上传落盘时，真实文件路径是这样拼的：

```c
test/files/<客户端传来的hash>
```

所以落盘文件名自然也是小写。

### 2.3.2 下载时真实文件名的来源

下载时，服务端不是用客户端再传一次 hash，而是：

1. 先根据 `paths` 找到 `file_id`
2. 再根据 `file_id` 到 `files` 表中查 `sha256sum`

问题出在这里：

在 MySQL 中，原来的代码是：

```sql
SELECT HEX(sha256sum), size FROM files WHERE id=...
```

`HEX()` 返回的是大写十六进制字符串。

所以最终拼出来的真实路径是：

```text
test/files/86D82221...
```

而磁盘上真实文件是：

```text
test/files/86d82221...
```

路径大小写不一致，`open()` 就失败了。

### 2.3.3 为什么这个问题只在联调时暴露

因为从静态代码看：

1. `files` 表确实存了 hash
2. 下载逻辑也确实能查到 hash
3. 路径拼接代码也没语法错误

但只有真正做到：

- 上传落盘
- 再下载打开

才会暴露“数据库输出格式”和“磁盘文件命名格式”不一致的问题。

这就是典型的“静态看不出来、联调才能发现”的问题。

---

## 2.4 修复方案

修复位置：

- [src/server/dao_file.c](/home/liwenshuo/my_project/WindCloud_V3/src/server/dao_file.c)

原来的 SQL：

```c
"SELECT HEX(sha256sum), size FROM files WHERE id=%d"
```

修复后的 SQL：

```c
"SELECT LOWER(HEX(sha256sum)), size FROM files WHERE id=%d"
```

也就是在数据库取值阶段，直接把结果统一转换成小写。

这样做的好处很直接：

1. 不用改磁盘文件命名规则
2. 不用改客户端 hash 上报规则
3. 只需要保证“下载阶段从数据库取出的 hash”与落盘文件名格式一致

修复后，下载时真实路径会变成：

```text
./test/files/86d82221f388c7a0...
```

于是 `open()` 成功，下载恢复正常。

---

## 2.5 修复后的结果

修复后再次联调：

客户端执行：

```text
gets windcloud_stage2_upload.txt
```

得到结果：

```text
下载成功: windcloud_stage2_upload.txt (58 字节)
```

并且重新计算下载后文件的 SHA-256，和原文件完全一致。

说明：

1. 数据库查询链路正确
2. 真实磁盘路径拼接正确
3. 下载数据内容也正确

---

## 2.6 以后如何避免同类问题

这个问题的本质，是“同一个业务字段在不同层的表示格式不统一”。

以后要避免类似问题，可以从下面几方面入手。

### 2.6.1 统一约定 hash 的表示格式

团队必须明确规定：

- 所有模块中，SHA-256 一律使用小写十六进制字符串

这样：

- 客户端上报小写
- 数据库存取也转成小写
- 磁盘文件名也用小写

整个系统表示统一，就不会再有这种问题。

### 2.6.2 为“格式转换点”写明确注释

凡是出现：

- `HEX()`
- `UNHEX()`
- 二进制转十六进制
- 十六进制转文件路径

这些地方，都应该写清楚：

```c
这里必须统一成小写，否则和磁盘落盘文件名不一致
```

### 2.6.3 联调时不要只看数据库，要同时看磁盘

这个问题就是因为：

- 数据库看起来没问题
- 但磁盘路径不对

所以以后遇到“文件存在但打开失败”这种问题时，必须同步检查：

1. 数据库里的值
2. 真正拼出来的路径
3. 磁盘上的实际文件名

### 2.6.4 对关键链路做完整测试

不能只测“上传成功”。

因为：

- 上传成功 ≠ 下载一定成功

上传和下载虽然都围绕同一份文件，但它们走的是不同代码路径。

以后这类功能至少要完整测：

1. 首次上传
2. 跨用户秒传
3. 下载
4. 下载后重新比对 hash

## 3. 问题二：同路径重复上传时客户端卡住

## 3.1 问题现象

在修完上传和秒传主流程后，我继续做更贴近真实使用场景的联调。

测试步骤是：

1. 用户 `stage2e` 登录
2. 成功上传 `windcloud_stage2_upload.txt`
3. 同一个用户在同一个目录里再次执行：

```text
puts windcloud_stage2_upload.txt
```

这时客户端表现异常：

- 终端停住不动
- 没有立即返回正常错误提示
- 看起来像“卡死了”

这个问题和前一个问题不一样。

前一个问题是“功能执行到最后失败”。

这个问题是“客户端和服务端对通信步骤的理解不一致，导致一方在等，另一方却没按这个顺序发”。

也就是典型的协议同步问题。

---

## 3.2 为什么会卡住

要理解这个问题，必须先看 `puts` 命令的正常协议流程。

客户端执行 `puts` 时，代码里默认认为服务端一定会按这个顺序回应：

1. 客户端发 `command_packet_t`
2. 客户端发 `file_packet_t`
3. 客户端等待服务端回一个 `file_packet_t`
4. 再根据这个 `file_packet_t` 决定：
   - 是秒传结束
   - 还是继续上传数据

也就是说，对客户端来说：

```text
服务端下一步一定先回 file_packet_t
```

这是一条协议约定。

问题就在于：

当服务端发现“同一路径下已经有这个文件名”时，原来的处理逻辑是：

```c
send_msg(client_fd, "错误：该文件已存在");
return;
```

也就是：

- 服务端直接发了普通文本响应包
- 没有先发客户端正在等待的 `file_packet_t`

于是两边状态就错位了：

### 客户端的想法

“我现在应该收到一个 `file_packet_t`，我要继续 `recv_file_packet()`。”

### 服务端的想法

“我已经发了一条错误消息了，这次请求结束了。”

结果就是：

- 客户端还在等文件包
- 服务端已经结束这次 `puts`

协议对不上，客户端自然就卡住。

---

## 3.3 根因分析

这个问题的根因不是“重名判断写错了”，而是：

```text
业务异常分支没有遵守 puts 协议的固定收发顺序
```

也就是说，正常路径考虑到了协议，异常路径却没考虑完整。

### 3.3.1 正常路径的协议是对的

正常上传时：

1. 客户端发命令包
2. 客户端发文件包
3. 服务端回文件包
4. 客户端继续上传数据

没问题。

### 3.3.2 秒传路径的协议也是对的

秒传时：

1. 客户端发命令包
2. 客户端发文件包
3. 服务端回文件包（其中 hash 相同）
4. 客户端据此判断“秒传成功”

也没问题。

### 3.3.3 唯独“重名拒绝”路径破坏了协议

在这个分支里，服务端跳过了“先回文件包”这一步。

所以这个问题本质上不是普通业务逻辑 bug，而是：

```text
协议分支不完整
```

### 3.3.4 为什么这个问题也只能通过联调发现

从代码表面看，服务端这样写似乎也讲得通：

```c
发现重名 -> 直接报错返回
```

如果只从单边代码看，很容易觉得“没问题啊”。

但只有把客户端和服务端放到一起跑，才会发现：

- 客户端根本不是在等文本包
- 客户端此时是在等文件包

所以这是一个典型的“双端协议联调问题”。

---

## 3.4 修复方案

修复位置：

- [src/server/file_transfer.c](/home/liwenshuo/my_project/WindCloud_V3/src/server/file_transfer.c)

修复思路不是简单地“多发一句提示”，而是：

```text
必须先把客户端正在等待的 file_packet_t 补齐，再发错误文本
```

修复后的逻辑是：

1. 构造一个 `file_packet_t`
2. 把 `offset` 设成 `client_file_packet.file_size`
3. 先把这个 `file_packet_t` 发给客户端
4. 再发一条普通文本：

```text
错误：该文件已存在
```

对应代码核心是：

```c
file_packet_t server_file_packet;
init_file_packet(&server_file_packet,
                 CMD_TYPE_PUTS,
                 arg,
                 client_file_packet.file_size,
                 client_file_packet.file_size,
                 NULL);

send_file_packet(client_fd, &server_file_packet);
send_msg(client_fd, "错误：该文件已存在");
```

这样做以后，协议顺序重新被拉回来了：

### 客户端看到的流程

1. `recv_file_packet()` 正常收到一个文件包
2. 然后客户端继续往后走
3. 收到普通文本错误消息
4. 把错误打印出来

客户端不再卡住。

---

## 3.5 为什么这个修法是正确的

因为协议问题的修复原则不是“想办法把错误告诉对方”，而是：

```text
即使是错误分支，也必须遵守双方之前约定好的收发顺序
```

这是网络程序里非常重要的一条原则。

正常情况和异常情况都应该满足：

- 双方什么时候该发什么
- 对方什么时候该收什么

只要有一边在某个异常分支“偷懒少发一步”，另一边就很容易挂住。

所以这次修复真正解决的不是“把提示补出来”，而是“把协议补完整”。

---

## 3.6 修复后的结果

修复后再次联调：

同一用户在同一路径重复上传同名文件，客户端结果变成：

```text
正在计算文件哈希值...
错误：该文件已存在
```

并且：

- 客户端不会卡住
- 服务端不会异常退出
- 连接协议保持正常

说明修复已经生效。

## 3.7 以后如何避免同类问题

这个问题本质上是“协议设计没有覆盖所有异常路径”。

以后要避免类似问题，必须重点抓下面几条。

### 3.7.1 先画清楚协议状态机

像 `puts`、`gets` 这种多阶段命令，不应该只在脑子里想“先发这个再发那个”，而应该明确写成状态机：

例如 `puts`：

1. 客户端发命令包
2. 客户端发文件包
3. 服务端必须回文件包
4. 客户端再决定下一步

只要协议状态机画清楚了，就能立刻看出：

- “重名拒绝”这个异常分支少了一步

### 3.7.2 所有异常分支都要对照协议检查

以后写网络代码，不要只检查“正常路径对不对”，还要检查：

1. 参数错误时发了什么
2. 文件不存在时发了什么
3. 重名时发了什么
4. 数据库失败时发了什么
5. 客户端应该在那一刻期待什么

### 3.7.3 文件传输类命令不要随意混用不同类型的回包

如果客户端当前阶段明确在等 `file_packet_t`，

那服务端就不应该突然改成只发 `command_packet_t`。

除非协议一开始就定义了“这里可能收到两种类型之一”，否则就必须保持一致。

### 3.7.4 联调时专门测异常场景

很多人联调时只测：

- 能不能成功上传
- 能不能成功下载

但网络程序真正容易出问题的，常常是这些异常路径：

1. 同名上传
2. 文件不存在
3. 登录失败
4. 断点位置异常
5. 数据库查询失败

以后这类命令必须明确加入“异常场景联调清单”。

## 4. 这两个问题带来的整体经验

这两个问题虽然表现不一样，但其实说明了同一个更深层的问题：

```text
系统开发不能只盯着单个模块代码是否“看起来合理”，必须盯整条链路是否真正闭环。
```

### 第一个问题说明

- 数据库里有值，不代表磁盘路径就一定对
- 字段值跨层传递时，格式一致性非常重要

### 第二个问题说明

- 服务端逻辑看起来能报错，不代表客户端就一定能正确接收
- 双端协议必须在正常和异常路径上都保持一致

换句话说：

### 对于存储链路

要检查：

```text
数据库值 -> 内存字符串 -> 真实路径
```

是否一致。

### 对于通信链路

要检查：

```text
客户端期待什么 -> 服务端是否真的按这个顺序发了什么
```

是否一致。

## 5. 后续开发建议

为了减少以后再出现类似问题，建议团队在后续开发时执行下面这些规则。

## 5.1 为关键字段建立统一规范表

例如专门列一个“全局字段表示规范”：

### `sha256sum`

- 客户端：小写十六进制字符串
- 协议：小写十六进制字符串
- 数据库查询输出：小写十六进制字符串
- 磁盘文件名：小写十六进制字符串

这样后面任何人再用这个字段，都不会各写各的。

## 5.2 为 `puts/gets/login` 等复杂命令写协议文档

不要只写“这个命令是干什么的”，而要写成：

```text
第 1 步谁发什么
第 2 步谁收什么
第 3 步谁再发什么
异常时谁仍然必须发什么
```

## 5.3 每次联调要覆盖“成功路径 + 异常路径”

建议每个命令至少测：

### `puts`

1. 首次上传
2. 同用户同路径重名上传
3. 跨用户同内容秒传
4. 断点续传

### `gets`

1. 正常下载
2. 本地已有部分文件续传
3. 请求不存在文件

### 登录

1. 正确密码
2. 错误密码
3. 不存在账号

## 5.4 遇到联调问题优先看“链路”，不要只看“单点”

以后团队里谁遇到类似问题，排查时优先画这两种链路：

### 存储链路

```text
输入 -> 数据库 -> 内存 -> 路径 -> 磁盘
```

### 通信链路

```text
客户端发送 -> 服务端接收 -> 服务端回应 -> 客户端接收
```

只要把链路画出来，问题通常就能更快定位。

---

## 6. 总结

这次联调发现并修复的两个问题，分别对应了网络程序开发里最容易被忽略的两类错误：

### 第一类：跨层格式不一致

表现：

- 数据库值、磁盘文件名、内存字符串不统一

后果：

- 看起来数据都有，最后一步却打不开文件

### 第二类：协议异常分支不完整

表现：

- 正常路径协议正确
- 异常路径少发或错发一个包

后果：

- 客户端阻塞
- 服务端和客户端状态错位

这两个问题最后都已经修复，而且联调也已经重新验证通过。
