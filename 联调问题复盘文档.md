# 本地联调核心问题复盘报告

## 1. 文档目的

本文档旨在对第二阶段本地联调过程中发现并修复的两个核心问题进行深度复盘：

1. **下载场景下的物理寻址失败问题**（逻辑文件存在，但真实文件读取失败）
2. **同路径重复上传时的客户端阻塞问题**（异常分支导致协议状态机错位）

这两类问题均非静态编译错误，而是只有在客户端、服务端、数据库及底层文件系统完整链路闭合后，才会暴露的系统级联调缺陷。

本文档的核心不仅在于记录“修改了哪行代码”，更在于系统性地梳理以下环节：

- 缺陷在外部交互层的具体表象。
- 静态逻辑看似正确，但在动态运行环境中失效的底层机制。
- 问题的根本原因（Root Cause）溯源。
- 最终确定的修复方案及其合理性。
- 为后续研发沉淀的架构规范与避坑指南。

------

## 2. 问题一：下载时物理文件寻址失败

### 2.1 缺陷现象

在本地联调的测试基准中，上传与秒传主流程均已验证通过（数据库中 `paths` 与 `files` 表记录均准确生成）。但在执行下载指令时发生异常：

测试账号 `stage2e` 执行下载：

```
gets windcloud_stage2_upload.txt
```

客户端收到错误回执：

```Plaintext
服务器文件不存在
```

**现象剖析**：

- 用户视角：文件已明确提示上传成功，但无法下载。
- 数据库视角：虚拟路径（`paths`）与文件元数据（`files`）记录完好。
- 系统视角：服务端在最终向文件系统发起物理文件读取（`open`系统调用）时遭遇失败。这表明核心缺陷位于“根据文件哈希值拼接物理路径”的最后一环。

------

### 2.2 排查链路与定位

针对此类问题，排查必须贯穿数据流转的全链路：

```
客户端请求 -> 虚拟路径匹配 -> 获取 paths.file_id -> 映射 files.sha256sum -> 拼接物理路径 -> 系统调用 open()
```

**第一步：核验数据库持久化状态**

经查验，数据库状态正常：

1. `paths` 表正确映射：`/windcloud_stage2_upload.txt` -> `file_id = 1`
2. `files` 表记录完整：`id = 1`, `sha256sum = 86D82221F388C7A0...`

**第二步：分析服务端运行日志**

服务端日志暴露了关键性错误：

```Plaintext
打开真实文件失败，路径=./test/files/86D82221F388C7A0B3933A6501013967E5B07B21A65CB9B9ED0487D45DF1A446，错误码=2 (ENOENT: No such file or directory)
```

比对物理磁盘上实际存在的文件名：

```Plaintext
./test/files/86d82221f388c7a0b3933a6501013967e5b07b21a65cb9b9ed0487d45df1a446
```

**定位结论**：

这是一起典型的因**大小写敏感（Case Sensitivity）**导致的文件寻址失败。数据库提取出的哈希值为大写，而磁盘上的物理文件名为小写。在 Linux 文件系统（如 ext4）中，两者被视为完全不同的文件。

------

### 2.3 根本原因分析

该缺陷的根因在于整个业务链路中，不同模块对 SHA-256 哈希值的数据格式缺乏统一的约束。

#### 2.3.1 物理文件名（小写）的来源

文件上传时，客户端通过 `sha256sum` 命令计算哈希值，默认生成**小写**十六进制字符串。服务端接收后直接以该字符串作为物理文件名进行持久化，因此磁盘文件名为小写。

#### 2.3.2 数据库提取值（大写）的来源

下载文件时，服务端的数据逆向检索逻辑为：

1. 依据 `paths` 匹配 `file_id`。
2. 依据 `file_id` 查询 `files` 表获取哈希值。

原系统使用的 SQL 语句为：

```sql
SELECT HEX(sha256sum), size FROM files WHERE id=...
```

MySQL 的 `HEX()` 函数默认返回**大写**十六进制字符串。因此，服务端拼接出的路径全为大写，最终导致 `open()` 失败。此类数据格式的不一致在单元测试和静态代码审查中极难被发现，是典型的集成环境隐患。

------

### 2.4 修复方案

**修复位置**：`src/server/dao_file.c`

**原本逻辑**：

```c
"SELECT HEX(sha256sum), size FROM files WHERE id=%d"
```

**优化逻辑**：

将原 SQL 修正为强制转小写输出：

```c
"SELECT LOWER(HEX(sha256sum)), size FROM files WHERE id=%d"
```

**方案优势**：

通过在数据流出层（DAO 层）统一格式，该方案做到了最小侵入性：无需改动底层物理文件的命名规则，也无需调整客户端的上报逻辑，仅从源头保证了“数据取值格式”与“物理存储格式”的绝对一致。

### 2.5 修复后的结果

修复后再次联调：

客户端执行：

```text
gets windcloud_stage2_upload.txt
```

得到结果：

```text
下载成功: windcloud_stage2_upload.txt (58 字节)
```

并且重新计算下载后文件的 SHA-256，和原文件完全一致。

说明：

1. 数据库查询链路正确
2. 真实磁盘路径拼接正确
3. 下载数据内容也正确

---

### 2.6 以后如何避免同类问题

这个问题的本质，是“同一个业务字段在不同层的表示格式不统一”。

以后要避免类似问题，可以从下面几方面入手。

#### 2.6.1 统一约定 hash 的表示格式

团队必须明确规定：

- 所有模块中，SHA-256 一律使用小写十六进制字符串

整个系统表示统一，就不会再有这种问题。

#### 2.6.2 为“格式转换点”写明确注释

凡是出现：

- `HEX()`
- `UNHEX()`
- 二进制转十六进制
- 十六进制转文件路径

这些地方，都应该写清楚：

```c
这里必须统一成小写，否则和磁盘实际文件名不一致
```

#### 2.6.3 联调时不要只看数据库，要同时看磁盘

这个问题就是因为：

- 数据库看起来没问题
- 但磁盘路径不对

所以以后遇到“文件存在但打开失败”这种问题时，必须同步检查：

1. 数据库里的值
2. 真正拼出来的路径
3. 磁盘上的实际文件名

#### 2.6.4 对关键链路做完整测试

不能只测“上传成功”。

因为：

- 上传成功 ≠ 下载一定成功

上传和下载虽然都围绕同一份文件，但它们走的是不同代码路径。

以后这类功能至少要完整测：

1. 首次上传
2. 跨用户秒传
3. 下载
4. 下载后重新比对 hash

## 3. 问题二：同路径重复上传导致客户端阻塞

### 3.1 缺陷现象

在核心流程跑通后，引入边界场景测试。当用户在同一目录下对同一文件重复执行上传指令时：

```Plaintext
puts windcloud_stage2_upload.txt
```

**外部表现**：客户端程序失去响应（挂起/假死），终端无输出，且未收到预期的服务端错误提示。

此问题并非常规的逻辑报错，而是典型的**网络通信协议错位**，即通信双方对当前所处状态机的理解产生了分歧。

------

### 3.2 阻塞原因溯源：协议状态机错位

在标准的 `puts` 指令协议中，双方约定的交互时序如下：

1. 客户端发送 `command_packet_t`（命令包）。
2. 客户端发送 `file_packet_t`（文件元数据包），**随后进入阻塞等待状态**。
3. 服务端必须回传一个 `file_packet_t` 作为确认。
4. 客户端依据收到的确认包，决定是执行“秒传结束”还是“启动数据块传输”。

也就是说，对客户端来说：

```text
服务端下一步一定先回 file_packet_t
```

这是一条协议约定。

**缺陷爆发点**：

当服务端检测到“路径冲突（同名文件已存在）”时，原有的异常处理逻辑为：

```c
send_msg(client_fd, "错误：该文件已存在");
return;
```

服务端直接抛出了一个普通的文本消息包，没有发送客户端正在等待的 `file_packet_t`，并单方面结束了当前会话。

**状态机冲突**：

- **客户端状态**：正阻塞在 `recv_file_packet()`，死等约定的 `file_packet_t` 结构体。

- **服务端状态**：发送文本包后，认为该指令已处理完毕。

  由于数据包类型与预期不符，客户端无法推进状态机，最终陷入死锁阻塞。

------

### 3.3 根本原因分析

该缺陷的本质是：**业务异常分支破坏了固有的网络协议状态机闭环**。

在网络编程中，正常路径与异常路径必须遵循同样的报文交互契约。服务端在处理冲突时，“越级”跳过了发送 `file_packet_t` 的步骤，导致协议链路断裂。这反映出在早期的协议设计中，未能充分覆盖所有的异常上下文。

#### 3.3.1 正常路径的协议是对的

正常上传时：

1. 客户端发命令包
2. 客户端发文件包
3. 服务端回文件包
4. 客户端继续上传数据

没问题。

#### 3.3.2 秒传路径的协议也是对的

秒传时：

1. 客户端发命令包
2. 客户端发文件包
3. 服务端回文件包（其中 hash 相同）
4. 客户端据此判断“秒传成功”

也没问题。

#### 3.3.3 唯独“重名拒绝”路径破坏了协议

在这个分支里，服务端跳过了“先回文件包”这一步。

所以这个问题本质上不是普通业务逻辑 bug，而是：

```text
协议分支不完整
```

#### 3.3.4 为什么这个问题也只能通过联调发现

从代码表面看，服务端这样写似乎也讲得通：

```c
发现重名 -> 直接报错返回
```

如果只从单边代码看，很容易觉得“没问题啊”。

但只有把客户端和服务端放到一起跑，才会发现：

- 客户端根本不是在等文本包
- 客户端此时是在等文件包

所以这是一个典型的“双端协议联调问题”。

------

### 3.4 修复方案与依据

**修复位置**：`src/server/file_transfer.c`

**修复策略**：

在抛出最终错误提示前，必须先“补发”客户端正在等待的协议报文，引导客户端状态机正常流转至下一步。然后再发送错误文本。

修复后的逻辑是：

1. 构造一个 `file_packet_t`
2. 把 `offset` 设成 `client_file_packet.file_size`
3. 先把这个 `file_packet_t` 发给客户端
4. 再发一条普通文本：

```text
错误：该文件已存在
```

**核心代码逻辑**：

```c
// 1. 构造一个占位的 file_packet_t 回执
file_packet_t server_file_packet;
init_file_packet(&server_file_packet,
                 CMD_TYPE_PUTS,
                 arg,
                 client_file_packet.file_size,
                 client_file_packet.file_size, // 关键：告知客户端无需再传数据
                 NULL);

// 2. 补齐协议报文，解除客户端阻塞
send_file_packet(client_fd, &server_file_packet);

// 3. 发送真实的业务错误信息
send_msg(client_fd, "错误：该文件已存在");
```

**修复依据**：

网络通信修复的核心原则并非简单地“将报错信息发出去”，而是“**确保即使在异常分支下，通信双方的时序和包类型依然符合协议契约**”。通过补齐回执包，客户端能正常完成接收并退出阻塞态，随后正确解析后续到达的文本错误包。

### 3.5 修复后的结果

修复后再次联调：

同一用户在同一路径重复上传同名文件，客户端结果变成：

```text
正在计算文件哈希值...
错误：该文件已存在
```

并且：

- 客户端不会卡住
- 服务端不会异常退出
- 连接协议保持正常

说明修复已经生效。

---

### 3.6 以后如何避免同类问题

这个问题本质上是“协议设计没有覆盖所有异常路径”。

以后要避免类似问题，必须重点抓下面几条。

#### 3.6.1 先画清楚协议状态机

像 `puts`、`gets` 这种多阶段命令，不应该只在脑子里想“先发这个再发那个”，而应该明确写成状态机：

例如 `puts`：

1. 客户端发命令包
2. 客户端发文件包
3. 服务端必须回文件包
4. 客户端再决定下一步

只要协议状态机画清楚了，就能立刻看出：

- “重名拒绝”这个异常分支少了一步

#### 3.6.2 所有异常分支都要对照协议检查

以后写网络代码，不要只检查“正常路径对不对”，还要检查：

1. 参数错误时发了什么
2. 文件不存在时发了什么
3. 重名时发了什么
4. 数据库失败时发了什么
5. 客户端应该在那一刻等待什么

#### 3.6.3 文件传输类命令不要随意混用不同类型的消息包

如果客户端当前阶段明确在等 `file_packet_t`，

那服务端就不应该突然改成只发 `command_packet_t`。

除非协议一开始就定义了“这里可能收到两种类型之一”，否则就必须保持一致。

#### 3.6.4 联调时专门测异常场景

很多人联调时只测：

- 能不能成功上传
- 能不能成功下载

但网络程序真正容易出问题的，常常是这些异常路径：

1. 同名上传
2. 文件不存在
3. 登录失败
4. 断点位置异常
5. 数据库查询失败

以后这类命令必须明确加入“异常场景联调清单”。

------

## 4. 总结与架构沉淀

这两个缺陷虽然表象不同，但都指向了系统集成开发中极易被忽视的核心盲区：**系统开发不能局限于单一模块的逻辑自洽，必须保障全链路、全状态机的绝对闭环**。

具体而言：

1. **存储链路（问题一）**：验证数据从“数据库提取 -> 内存转换 -> 物理路径拼接 -> 磁盘寻址”的格式一致性。
2. **通信链路（问题二）**：验证双方在“正常收发”与“异常阻断”下的协议状态机是否永远保持同步。

------

## 5. 后续研发规范建议

为避免类似缺陷在后续迭代中再次出现，建议团队推行以下研发规范：

### 5.1 建立全局核心字段的数据字典

针对类似 `sha256sum` 这种跨越客户端、网络层、数据库和文件系统的核心业务字段，必须在设计文档中明确定义其全局唯一的格式标准。

*示例：* `sha256sum` 全局统一规范为 **小写十六进制字符串**。

### 5.2 绘制并严格遵循协议状态机图

针对如 `puts`、`gets` 等多阶段复合指令，摒弃口头约定的开发模式，必须输出标准的时序图（Sequence Diagram）或状态机图。明确规定在每一个阶段，双方各自期望接收的包类型。当进入业务异常分支（如权限不足、文件冲突）时，必须对照状态机图，确保发出的退回报文符合当前状态的要求。

### 5.3 强化边界与异常场景的联调覆盖

测试用例设计应大幅增加对异常链路的覆盖度，将以下场景纳入“提测前必查清单”：

- **复合指令边界**：同名覆盖、目标路径不存在、空文件传输。
- **状态机异常**：传输中途强制断开、非法包类型注入。
- **存储边界**：底层目录缺失、数据库脏数据检索。

## 6. 总结

这次联调发现并修复的两个问题，分别对应了网络程序开发里最容易被忽略的两类错误：

### 第一类：跨层格式不一致

表现：

- 数据库值、磁盘文件名、内存字符串不统一

后果：

- 看起来数据都有，最后一步却打不开文件

### 第二类：协议异常分支不完整

表现：

- 正常路径协议正确
- 异常路径少发或错发一个包

后果：

- 客户端阻塞
- 服务端和客户端状态错位
