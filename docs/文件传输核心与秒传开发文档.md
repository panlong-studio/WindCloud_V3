# 文件传输核心与秒传开发文档

## 1. 文档目标

本文档只说明成员 E 负责的“文件传输核心与秒传”部分，重点回答下面几个问题：

1. 为什么现在不能再按“用户虚拟路径 = 服务器真实路径”直接存文件。
2. `files` 表、`paths` 表、真实磁盘文件三者之间到底是什么关系。
3. 上传时如何判断“秒传”还是“真的收文件”。
4. 下载时如何根据用户当前目录，把真正的物理文件找出来。
5. 本次改动新增了哪些接口、这些接口分别做什么。

---

## 2. 先理解整体设计

第三期改造之后，服务器不再把用户文件直接按名字丢到用户目录里。

现在系统分成两层：

- 逻辑层：用户看到的目录树，保存在 `paths` 表里
- 物理层：服务器真正保存的文件，保存在 `test/files/` 目录里

### 2.1 逻辑层

逻辑层站在“用户视角”看问题。

例如用户 `zs` 当前在 `/doc` 下上传了一个 `a.txt`，那么用户以为这个文件存在于：

```text
/doc/a.txt
```

这条关系会存到 `paths` 表中，大致是：

| 字段 | 值 |
| --- | --- |
| `user_id` | `zs` 的 id |
| `path` | `/doc/a.txt` |
| `parent_id` | `/doc` 这个目录节点的 id |
| `file_name` | `a.txt` |
| `type` | `0` |
| `file_id` | 指向 `files` 表中的真实文件 id |

### 2.2 物理层

物理层站在“服务器实现者视角”看问题。

同内容文件，不管哪个用户上传，不管用户给它起什么名字，服务器都只保留一份真实文件。

这份真实文件的信息存在 `files` 表里：

| 字段 | 含义 |
| --- | --- |
| `id` | 真实文件主键 |
| `sha256sum` | 文件内容的 SHA-256 |
| `size` | 文件大小 |
| `count` | 被多少个逻辑文件引用 |

真实磁盘文件的名字，不再使用用户原始文件名，而是直接使用 SHA-256：

```text
test/files/<sha256值>
```

例如：

```text
test/files/4c26b435423089d46e5e40b0b5e0c1f1f0c5a0b3f7d8d8d0b2f4d0e9c1a2b3c4
```

这样有两个直接好处：

1. 相同内容文件天然去重
2. 下载时只要查到 hash，就一定能定位到真实文件

## 3. 上传时的核心思路

客户端在执行 `puts a.txt` 时，会先计算本地文件的 SHA-256，然后把它放进 `file_packet_t.hash` 发给服务端。

服务端收到后，分两种情况：

### 3.1 情况一：`files` 表里已经有这个 SHA-256

说明服务器以前已经收过一份完全相同的文件。

那么这次就不需要再进入真正的 `recv` 收文件循环，直接：

1. 在 `paths` 表里插入一条新的逻辑文件记录
2. 让 `files.count = files.count + 1`
3. 给客户端回一个 `file_packet_t`
4. 回包里的 `hash` 填客户端发来的同一个 SHA-256

客户端发现：

```c
strcmp(server_file_packet.hash, file_hash) == 0
```

就会直接判定为“极速秒传成功”。

### 3.2 情况二：`files` 表里没有这个 SHA-256

说明服务器还没有这个真实文件。

此时就进入正常上传流程：

1. 把真实文件路径拼成 `test/files/<sha256>`
2. 打开这个真实文件
3. 看它当前已经有多少字节
4. 把这个偏移量回给客户端，继续走断点续传
5. 文件收完整后，再往 `files` 表插一条记录
6. 再往 `paths` 表插一条逻辑文件记录

这里保留了一个非常适合教学项目的简单设计：

- 真实文件传完之前，不插入 `files` 表
- 但是磁盘上允许保留半截文件
- 下次上传同一份文件时，仍然可以根据同一个 hash 接着续传

这样实现简单，而且和原有断点续传逻辑兼容。

## 4. 下载时的核心思路

下载不再走“当前路径直接拼真实磁盘路径”的旧逻辑。

现在的流程是：

1. 先把 `当前虚拟路径 + 文件名` 拼成逻辑路径
2. 去 `paths` 表查这条逻辑路径，拿到 `file_id`
3. 再去 `files` 表查这个 `file_id` 对应的 `sha256sum`
4. 把真实磁盘路径拼成 `test/files/<sha256>`
5. 打开真实文件并 `sendfile`

所以下载链路是：

```text
用户输入文件名
-> 当前虚拟路径
-> paths.path
-> paths.file_id
-> files.sha256sum
-> test/files/<sha256>
-> sendfile
```

## 5. 本次新增/修改的关键接口

## 5.1 `dao_file.h` / `dao_file.c`

这一组接口只负责操作 `files` 表。

### `dao_file_find_by_sha256()`

作用：

- 根据 SHA-256 查真实文件是否已经存在
- 查到后返回 `file_id` 和 `size`

对应 SQL：

```sql
SELECT id, size FROM files WHERE sha256sum = UNHEX('...')
```

这里之所以用 `UNHEX()`，是因为表里 `sha256sum` 类型是 `binary(32)`，而客户端发来的是 64 位十六进制字符串。

### `dao_file_insert()`

作用：

- 在 `files` 表插入一条新的真实文件记录

对应 SQL：

```sql
INSERT INTO files (sha256sum, size, count)
VALUES (UNHEX('...'), 文件大小, 1)
```

### `dao_file_add_ref_count()`

作用：

- 秒传时把引用计数加 1

对应 SQL：

```sql
UPDATE files SET count = count + 1 WHERE id = ?
```

### `dao_file_get_info_by_id()`

作用：

- 下载时根据 `file_id` 反查真实文件的 SHA-256

对应 SQL：

```sql
SELECT HEX(sha256sum), size FROM files WHERE id = ?
```

这里用 `HEX()` 是为了把数据库里的二进制 hash 转回字符串。

## 5.3 `handle_puts()` 现在的完整流程

服务端版 `handle_puts()` 现在可以分成 6 步理解：

### 第 1 步：先收客户端发来的 `file_packet_t`

这一步必须最先做。

因为客户端在发送 `puts` 命令后，马上就会继续发文件信息结构体。

如果服务端不先把这个结构体读走，协议流就会错位。

### 第 2 步：拼逻辑路径，检查重名

例如当前路径是 `/doc`，用户上传 `a.txt`，则逻辑路径就是：

```text
/doc/a.txt
```

再查 `paths` 表，看这个用户当前目录里是不是已经有同名文件了。

### 第 3 步：看 hash 是否已存在

如果 `dao_file_find_by_sha256()` 查到了记录：

- 直接插 `paths`
- `count + 1`
- 回一个带同 hash 的 `file_packet_t`
- 返回

这就是秒传。

### 第 4 步：如果 hash 不存在，就定位真实磁盘文件

真实磁盘文件路径不是用户文件名，而是：

```text
test/files/<sha256>
```

### 第 5 步：继续用断点续传接收文件

这一部分仍然延续原项目已经有的断点续传思想：

- 服务端先看磁盘上已有多少字节
- 把偏移量发给客户端
- 客户端从这个偏移量继续发送
- 服务端用 `mmap + recv` 写入剩余数据

### 第 6 步：文件完整后补数据库

上传真正完成后，再执行：

1. 插入 `files`
2. 插入 `paths`

如果刚好遇到并发场景，别的线程已经先插入了同一个 hash：

- 当前线程会退化成“只补 `paths` + `count+1`”

## 5.4 `handle_gets()` 现在的完整流程

### 第 1 步：拼逻辑路径

当前目录 `/doc`，下载 `a.txt`：

```text
/doc/a.txt
```

### 第 2 步：查 `paths` 表

拿到这个逻辑文件对应的 `file_id`。

### 第 3 步：查 `files` 表

拿到：

- `sha256sum`
- `size`

### 第 4 步：拼真实路径

```text
test/files/<sha256>
```

### 第 5 步：回文件大小 + 收客户端断点位置

这一步和旧版下载断点续传逻辑一致。

### 第 6 步：`sendfile`

继续使用原项目里已经存在的高效发送方式。

## 6. 真实文件目录为什么要统一到 `test/files/`

如果还按旧思路把文件放在用户目录里，会出现几个问题：

1. 同一份内容会保存很多份
2. 秒传没有落点，因为根本没有“全局唯一真实文件”
3. 下载时找文件要依赖用户目录结构，不利于后续数据库化

统一到一个目录后：

- 目录关系交给 `paths`
- 真实文件去重交给 `files`
- 物理落盘只看 hash

职责非常清楚。

---

## 7. 为什么仍然保留断点续传

秒传和断点续传解决的是两个不同问题：

- 秒传：服务器已经有完整同内容文件，根本不收
- 断点续传：服务器还没有完整文件，但磁盘上可能已经收过一部分

所以正确关系不是“有了秒传就不要断点续传”，而是：

```text
先判断能不能秒传
不能秒传，再进入正常上传/断点续传
```

## 9. 关键代码片段说明

### 9.1 秒传判断

```c
if (dao_file_find_by_sha256(client_file_packet.hash, &existed_file_id, &existed_file_size) == 0) {
    create_user_file_link(...);
    init_file_packet(&server_file_packet, CMD_TYPE_PUTS, arg,
                     client_file_packet.file_size, 0, client_file_packet.hash);
    send_file_packet(client_fd, &server_file_packet);
    return;
}
```

这段代码的核心点：

- 如果数据库里已经有 hash，不进入收文件循环
- 回包时把同一个 hash 再发给客户端
- 客户端据此判定为秒传成功

### 9.2 真实文件路径生成

```c
snprintf(real_path, size, "%s/%s", store_dir, sha256sum);
```

这里非常关键。

以后真实文件名不再是 `a.txt`、`电影.mp4` 这种原名，而是直接用 hash。

### 9.3 下载时的双表反查

```c
dao_get_file_info_by_path(ctx->user_id, full_path, &node_id, &file_id);
dao_file_get_info_by_id(file_id, sha256sum, &file_size);
```

第一句从 `paths` 找逻辑节点。

第二句从 `files` 找真实文件。

这正是“逻辑路径”和“物理文件”解耦后的典型写法。

## 10. 当前实现的边界

为了保持这份 C 初学者项目代码尽量简单直观，本次实现刻意没有引入复杂工程化设计。

当前边界如下：

1. `puts/gets` 当前按“当前目录下的单层文件名”处理，不支持带 `/` 的多级相对路径参数。
2. 没有引入数据库事务，因此“插 `paths`、改 `count`”不是严格原子操作。
3. 没有专门设计“上传中”数据表，而是直接利用 hash 命名的半截物理文件继续做断点续传。

这些都是教学项目里完全可以接受的取舍。

## 11. 一句话总结

成员 E 这一部分的核心思想可以浓缩成一句话：

```text
上传时先查 hash，能秒传就只写数据库，不能秒传就按 hash 落真实文件；
下载时先查逻辑路径，再查真实文件 hash，最后按 hash 找物理文件发送。
```

只要把这句话真正理解了，整个“文件传输核心与秒传”模块就掌握了。
