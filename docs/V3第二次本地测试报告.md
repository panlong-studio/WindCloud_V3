- 下载续传：预先在客户端本地写入一个半截文件

这样可以稳定验证续传协议与偏移恢复逻辑。

## 6. 发现的缺陷

## 缺陷 1：删除文件后 `files.count` 不会递减

### 现象

用户 `e2ea145340` 已经删除了自己目录中的：

- `touched145340.txt`
- `e2e_small_20260422_145340.txt`

并且目录 `dir145340` 也已经成功删除。

但数据库里：

- 小文件 hash 的 `count` 仍然是 `2`
- 空文件 hash 的 `count` 仍然是 `2`

而此时 `paths` 里只剩用户 `e2eb145340` 的一条对应逻辑文件。

### 影响

1. `files.count` 会越来越不准确
2. 以后即使某个真实文件已经没有逻辑引用，也无法安全回收
3. `touch` 与秒传路径会继续累计错误计数

### 结论

这是一个**功能正确性缺陷**，当前删除链路没有把 `paths` 删除与 `files.count` 生命周期绑定起来。

---

## 缺陷 2：超长文件名上传会留下不一致状态

### 复现条件

上传文件名：

```text
e2e_resume_fast_20260422_145340.bin
```

文件名长度：

```text
35
```

而数据库字段：

```text
paths.file_name VARCHAR(30)
```

### 现象

客户端输出：

```text
正在计算文件哈希值...
上传失败：数据库写入失败
```

但随后核验发现：

1. 对应 hash 的物理文件已经被补全
2. `files` 表中已经存在该 hash 记录
3. 这条长文件名本身却没有成功落到 `paths`

### 影响

1. 客户端看到的是上传失败
2. 服务端实际上已经产生了物理文件和 `files` 元数据
3. 会留下“孤儿真实文件 / 孤儿 files 记录”

### 结论

这是一个**数据库一致性缺陷**。当前应该在客户端或服务端更早做文件名长度校验，或者在数据库失败时做回滚。

---

## 缺陷 3：秒传只信任 `files` 表，不校验物理文件是否完整

### 复现方式

1. 让数据库中存在 hash `9bcf...` 的 `files` 记录
2. 人为把物理文件 `test/files/9bcf...` 截断为 **8MB**
3. 再用同 hash 文件执行上传

### 现象

客户端直接显示：

```text
极速秒传成功。
```

数据库也新增了逻辑路径：

```text
/resume145340/rs145340.bin
```

但磁盘上的真实文件大小实际上只有：

```text
hash_9bcf_disk_size=8388608
```

而 `files` 表仍记录：

```text
sha256sum=9bcf...
size=67108864
count=2
```

### 影响

1. 逻辑文件节点会指向损坏的物理文件
2. 下载时客户端可能把 8MB 的残缺文件误认为“完整文件”
3. `files` 表与真实磁盘状态脱节后，秒传会继续放大错误

### 结论

这是一个**高优先级一致性缺陷**。秒传分支当前只检查数据库，不检查磁盘实体是否存在且长度正确。

## 7. 最终结论

从“功能主链路”看，当前项目已经具备较完整的可用性：

- 登录 / 注册：可用
- 目录浏览与切换：可用
- `touch` 空文件创建与下载：可用
- 小文件上传 / 下载：可用
- 跨用户秒传：可用
- 上传续传 / 下载续传：可用

但从“一致性与边界条件”看，当前仍有 3 个必须正视的问题：

1. 删除链路没有维护 `files.count`
2. 超长文件名失败时没有做完整回滚
3. 秒传没有校验物理文件完整性

因此当前版本更准确的评价是：

> 基础功能已经跑通，主流程可用，但数据一致性和部分边界防护仍需补强。

## 8. 建议的修复优先级

### P1

1. `rm` / `rmdir` 相关删除链路补上 `files.count` 递减逻辑
2. 当 `count == 0` 时回收 `files` 记录和物理文件

### P1

1. 上传前统一校验文件名长度
2. 防止超过 `paths.file_name` 上限后才在数据库阶段失败
3. 数据库失败时补回滚

### P1

1. 秒传前除了查 `files` 表，还应检查 `test/files/<hash>` 是否存在
2. 并校验物理文件大小是否与 `files.size` 一致
3. 如果不一致，应走补传 / 修复路径，而不是直接秒传
