# 本地测试缺陷修复复盘文档

## 1. 文档目的

本文档用于复盘 2026-04-22 本地全功能与边界测试中暴露出的 3 个真实缺陷：

1. `rm` 删除逻辑文件后，`files.count` 不递减，真实文件也不会被回收。
2. 超长文件名在上传失败时可能遗留 `files` 表记录或磁盘实体，造成脏数据。
3. 秒传只信任 `files` 表，不校验 `test/files/<sha256>` 是否真实完整，可能把坏实体继续复用给新逻辑节点。

这 3 个问题都不是“编译错误”或“单函数逻辑错误”，而是数据库、虚拟路径、真实文件仓库、客户端协议协同之后才会暴露出来的链路一致性问题。

本文档重点说明：

- 缺陷在联调时的外部表现
- 根因分析
- 最终修复方案
- 本地回归验证结果
- 后续应固化的工程约束

---

## 2. 问题一：删除文件后引用计数不回收

### 2.1 缺陷现象

在原实现中，`rm <文件名>` 只删除了 `paths` 表中的逻辑节点：

- 用户目录里看起来文件已经消失
- 但 `files.count` 仍保持原值
- 当最后一个逻辑引用被删除后，`files` 表记录和 `test/files/<sha256>` 真实文件仍然残留

这会导致两个直接后果：

1. `files.count` 逐步失真，不再反映真实引用关系。
2. 磁盘实体无法回收，长期运行后会不断累积孤儿文件和无效元数据。

---

### 2.2 根因分析

原 `handle_rm()` 的删除链路只覆盖了虚拟文件系统这一层：

```text
客户端 rm -> 依据 paths.id 删除逻辑节点 -> 返回“删除成功”
```

但真实文件这一层没有参与：

```text
paths 删除完成
files.count 不变
test/files/<sha256> 不处理
```

这说明当时的删除逻辑只把“文件”理解成了用户目录中的一个路径节点，没有把它看成：

1. `paths` 里的逻辑引用
2. `files` 里的真实实体
3. `test/files` 中的真实物理文件

本质上，这是删除链路缺少“引用计数回收”和“物理文件回收”两个步骤。

---

### 2.3 修复方案

修复落在以下几个位置：

- [include/dao_file.h](/home/liwenshuo/my_project/WindCloud_V3/include/dao_file.h)
- [src/server/dao_file.c](/home/liwenshuo/my_project/WindCloud_V3/src/server/dao_file.c)
- [src/server/file_cmds.c](/home/liwenshuo/my_project/WindCloud_V3/src/server/file_cmds.c)

具体改动如下：

1. 新增 `dao_file_sub_ref_count()`，负责把 `files.count` 减 1。
2. 新增 `dao_file_get_ref_count()`，用于删除后判断该真实文件是否还有剩余逻辑引用。
3. 新增 `release_file_entity_if_unused()`，把“引用计数递减 + 归零时回收物理文件 + 删除 files 记录”封装成一条删除后链路。
4. `handle_rm()` 在删除 `paths` 节点前，先根据逻辑路径查出对应的 `file_id`，再在删除逻辑节点成功后触发真实文件回收。

现在的删除链路变为：

```text
客户端 rm
-> 查询目标逻辑路径对应 file_id
-> 删除 paths 节点
-> files.count 减 1
-> 若 count > 0：保留真实文件
-> 若 count == 0：删除 test/files/<sha256> 与 files 表记录
```

这里额外补了注释，明确说明：

- 为什么删除逻辑文件时必须同步维护 `files.count`
- 为什么只有 `count == 0` 才能回收磁盘实体

---

### 2.4 回归验证结果

本地回归分两组验证。

#### 验证 A：单引用删除

测试过程：

1. 用户 `fix422a` 上传 `verify_fix_small.txt`
2. 执行 `rm verify_fix_small.txt`
3. 查询数据库与磁盘

验证结果：

- `paths` 中 `/verify_fix_small.txt` 已删除
- `files` 中对应 hash 记录已删除
- `test/files/39ad983c6f00e2e328b101cd6f23c1ca1b8dac08f2690d76fe3225ddab98688a` 已不存在

说明“最后一个引用删除后，真实文件会被完整回收”。

#### 验证 B：多引用删除

测试过程：

1. 用户 `fix422b` 上传 `verify_fix_small.txt`
2. 用户 `fix422c` 上传同内容文件，触发秒传复用
3. 查询 `files.count = 2`
4. 用户 `fix422b` 删除该文件
5. 查询 `files.count = 1` 且真实文件仍存在
6. 用户 `fix422c` 再删除该文件
7. 查询 `files` 记录消失，真实文件被回收

验证结果：

- 中间态 `count` 从 `2` 正确降为 `1`
- 最终态 `count` 归零后，数据库记录和物理文件均被清理

这说明删除链路现在已经满足“共享实体引用计数”的正确语义。

---

### 2.5 经验总结

以后任何“删除文件”需求，都不能只删目录项，必须明确它删的是哪一层：

1. 逻辑路径层
2. 真实文件元数据层
3. 物理文件层

如果系统采用“去重存储 + 引用计数”模型，那么删除流程必须被设计成一条完整事务语义链，而不是只做 `DELETE FROM paths`。

## 3. 问题二：超长文件名导致脏数据风险

### 3.1 缺陷现象

数据库中 `paths.file_name` 的定义是：

```sql
varchar(30)
```

但原实现中，客户端和服务端都没有在进入数据库前统一校验文件名长度。结果是：

- 客户端可以发起超长文件名上传
- 服务端可能已经创建了真实文件或插入了 `files` 记录
- 直到插入 `paths` 时才被数据库拒绝
- 最终表现为“上传失败”，但 `files` 表或磁盘实体已经留下脏数据

这类问题最危险的地方在于：

- 用户看到的是一次失败上传
- 但系统内部已经写入了部分状态
- 这些状态不会自动回滚

---

### 3.2 根因分析

根因并不在 SQL 本身，而在“输入约束下沉得太晚”。

原来的处理链路大致是：

```text
客户端 puts
-> 服务端创建 / 复用真实文件
-> 服务端写 files 表
-> 服务端写 paths 表
-> 因 file_name > 30 被数据库拒绝
```

也就是说，系统把“文件名是否合法”这个判断交给了数据库尾端来兜底，而不是在业务入口就拦截。

这种设计会带来两个问题：

1. 错误暴露得太晚，前置副作用已经发生。
2. 客户端和服务端对名字合法性的认知不一致，容易形成协议层和存储层脱节。

---

### 3.3 修复方案

修复点位于：

- [include/protocol.h](/home/liwenshuo/my_project/WindCloud_V3/include/protocol.h)
- [src/client/client_command_handle.c](/home/liwenshuo/my_project/WindCloud_V3/src/client/client_command_handle.c)
- [src/server/file_cmds.c](/home/liwenshuo/my_project/WindCloud_V3/src/server/file_cmds.c)
- [src/server/file_transfer.c](/home/liwenshuo/my_project/WindCloud_V3/src/server/file_transfer.c)

本次采取的是“客户端提前拦截 + 服务端强制兜底”的双层约束。

#### 3.3.1 统一长度常量

在 `protocol.h` 中新增：

```c
#define MAX_VFS_NAME_LEN 30
```

这样客户端和服务端共享同一个数据库约束，不再各写各的魔法数字。

#### 3.3.2 客户端前置校验

客户端现在会对以下命令参数做长度检查：

- `puts`
- `gets`
- `mkdir`
- `touch`
- `rm`
- `rmdir`

超过 30 个字符时，直接提示：

```text
文件名或目录名过长，最大长度为 30
```

这样绝大多数非法输入在发包之前就被拦下，不再进入服务端。

#### 3.3.3 服务端兜底校验

服务端在 `handle_puts()`、`handle_touch()`、`handle_mkdir()` 中也增加了同样的名称校验。

这样即便未来存在：

- 非官方客户端
- 手工构造协议包
- 客户端校验被绕过

服务端仍然会在进入数据库之前拒绝该请求，避免再写出半成品状态。

---

### 3.4 回归验证结果

本地使用账号 `fix422a` 进行了超长文件名测试，文件名为：

```text
1234567890123456789012345678901.txt
```

验证命令：

```text
puts 1234567890123456789012345678901.txt
touch 1234567890123456789012345678901.txt
mkdir 1234567890123456789012345678901.txt
```

客户端结果：

```text
文件名或目录名过长，最大长度为 30
```

数据库结果：

- `files` 总数未增加
- `paths` 总数未增加
- `paths.file_name = '1234567890123456789012345678901.txt'` 的记录不存在

这说明超长文件名已经被拦在业务入口，不会再把状态推进到数据库写入阶段。

---

### 3.5 经验总结

凡是数据库字段存在明确上限的输入，都不应该把约束延后到 SQL 报错阶段再暴露。

以后需要固化两条规则：

1. 约束常量统一定义在公共头文件，避免客户端、服务端、数据库三边不一致。
2. 所有“可能产生副作用”的业务入口，都要在副作用发生前完成校验。

## 4. 问题三：秒传错误复用损坏的真实文件

### 4.1 缺陷现象

原实现的秒传判断逻辑是：

```text
只要 files 表里存在同 hash 记录
=> 直接认为服务器已经有完整真实文件
=> 为新用户新建逻辑节点
=> 返回秒传成功
```

这在“数据库正确、磁盘完整”的理想状态下没有问题，但一旦真实文件残缺，就会出现严重错误：

1. 数据库 `files` 表有该 hash
2. `test/files/<sha256>` 缺失，或者只剩一部分
3. 新用户上传同内容文件时被直接秒传
4. 新逻辑路径其实指向了一份损坏的真实文件

后续一旦下载，就可能得到半截文件或直接失败。

---

### 4.2 根因分析

这个问题的根因是秒传判定条件不完整。

系统之前把“秒传成立”简化成了：

```text
files 表存在 hash 记录
```

但实际上，秒传成立的前提至少应该同时满足：

1. `files` 表中存在该 hash 记录
2. `test/files/<sha256>` 真实文件存在
3. 真实文件大小与 `files.size` 一致

缺少第 2 和第 3 条，就意味着系统把“数据库元数据存在”误判成了“真实文件一定可用”。

这本质上是把“元数据一致性”错当成了“实体一致性”。

---

### 4.3 修复方案

修复位置在 [src/server/file_transfer.c](/home/liwenshuo/my_project/WindCloud_V3/src/server/file_transfer.c)。

本次新增了 `check_store_file_ready()`，在真正进入秒传分支前，先核验：

1. `test/files/<sha256>` 是否存在
2. 是否为普通文件
3. 文件大小是否等于 `files.size`

现在秒传判断逻辑变成：

```text
files 表命中 hash
-> 检查真实文件是否完整
-> 若完整：秒传
-> 若缺失 / 大小不一致：转为正常上传或断点续传
```

这样做的好处是：

- 数据库记录和真实文件不一致时，不会再把坏实体继续扩散给新的逻辑节点
- 如果真实文件只是“上传中断后残留半截”，系统还能复用已有偏移继续补传，而不是全量重来

文中对应代码也补了注释，明确写清楚：

- 秒传不能只依赖 `files` 表
- 真实文件不完整时必须降级为正常上传 / 续传

---

### 4.4 回归验证结果

本地回归步骤如下。

#### 第一步：先生成一份完整真实文件

用户 `fix422a` 上传 `verify_fix_partial.txt`，其 hash 为：

```text
9f1dcbc35c350d6027f98be0f5c8b43b42ca52b7604459c0c42be3aa88913d47
```

初始状态：

- `files.size = 8192`
- `files.count = 1`
- 磁盘文件大小 = `8192`

#### 第二步：人为制造半截物理文件

将对应真实文件截断为 `3000` 字节。

此时状态变成：

- `files.size = 8192`
- 磁盘文件大小 = `3000`

这正是“数据库有记录，但真实文件不完整”的故障现场。

#### 第三步：再次上传同内容文件

用户 `fix422d` 上传同一个 `verify_fix_partial.txt`。

修复前的错误行为应该是：

```text
极速秒传成功。
```

修复后的实际行为是：

```text
上传完成！
```

服务端日志中出现关键告警：

```text
检测到真实文件缺失或不完整，转为正常上传
```

最终状态：

- `files.count = 2`
- `test/files/<sha256>` 文件大小恢复为 `8192`
- 两个用户的逻辑路径都正确指向同一个完整实体

这说明秒传分支已经不会错误复用坏实体，而是会自动降级并修复真实文件。

---

### 4.5 经验总结

任何“命中元数据就跳过数据传输”的优化，本质上都属于一致性敏感逻辑。

以后设计这类优化时，必须区分两件事：

1. 元数据是否存在
2. 实体数据是否真的可用

如果没有对实体做校验，优化分支就可能变成错误扩散器，而不是性能优化点。

---

## 5. 整体修复结论

本次 3 个缺陷虽然表现不同，但本质都属于同一类问题：

**系统把某一层的状态误当成了“全链路状态已经正确”。**

具体表现为：

- 删除时，只看 `paths`，没把 `files` 和物理文件纳入同一条链路。
- 上传命名时，只依赖数据库尾端约束，没在业务入口提前阻断。
- 秒传时，只看 `files` 表记录，没校验真实文件是否仍然完整。

本轮修复后，系统在三个关键方向上都补齐了“链路闭合”：

1. 删除链路从“删逻辑节点”升级为“删逻辑节点 + 维护引用计数 + 回收实体”。
2. 文件名校验从“SQL 尾端兜底”升级为“客户端前置校验 + 服务端强制兜底”。
3. 秒传判定从“只看数据库”升级为“数据库 + 真实文件一致性联合校验”。

---

## 6. 后续建议

结合这次复盘，后续建议继续补充以下工程约束：

### 6.1 为关键链路建立固定回归用例

建议把下面 3 组场景固化成回归清单：

1. 多用户共享文件后逐次删除，检查 `count` 和真实文件回收。
2. 边界长度文件名输入，检查客户端提示和数据库无脏数据。
3. 人为制造半截真实文件，再次上传同 hash 文件，检查是否降级为续传。

### 6.2 把“副作用前校验”作为统一开发规范

所有可能写数据库、落磁盘、修改会话状态的代码，都应先做参数与环境校验，再做副作用操作。

### 6.3 对“数据库状态”和“磁盘状态”做联合思维

该项目已经不是单纯的数据库项目，也不是单纯的文件传输项目，而是一个“数据库元数据 + 虚拟文件系统 + 真实文件仓库”三者耦合的系统。

因此以后排查问题时，不能只看一层：

1. 客户端看到什么
2. 服务端日志记录了什么
3. 数据库是什么状态
4. `test/files` 中的真实文件是什么状态

只有这四层一起看，才能真正定位这类联调缺陷。
